# Architecture Decision Records

## 철학

공부기록 + 포트폴리오를 오래 편하게 운영하는 것이 목표. **유지보수 간편함과 확장성**을 우선한다. 콘텐츠는 마크다운으로 단순하게, 사이트는 정적으로 안전하게. 작성자는 한국어만 쓰고, 나머지(영문 번역·카테고리 노출·배포)는 자동화한다.

---

### ADR-001: 프레임워크로 Astro 선택
**결정**: Jekyll(academicpages) / Hugo 대신 **Astro 7**.
**이유**: 요구사항(한/영 이중언어, 중첩 카테고리, 자유로운 마크다운, 향후 확장)에 균형이 가장 좋다. npm 생태계로 확장성이 뛰어나고, Content Collections 스키마 검증으로 유지보수가 견고하다. 현재 로컬 환경(Node 22)에서 바로 동작.
**트레이드오프**: GitHub Pages 기본 지원(Jekyll)이 아니라 GitHub Actions 빌드가 필요하고, npm 의존성을 가끔 업데이트해야 한다. → Actions가 설치·빌드·배포를 자동 처리하므로 체감 부담은 작다.

### ADR-002: 한/영 전환을 URL이 아닌 클라이언트 토글로 구현
**결정**: 로케일별 URL(`/en/…`)을 만들지 않고, **언어중립 URL 1개**에 한/영 본문을 모두 렌더한 뒤 `<html data-lang>` + CSS로 활성 언어만 표시. `localStorage`로 언어 유지, `<head>` 인라인 스크립트로 FOUC 방지.
**이유**: "URL에 `ko`/`en`이 보이면 안 되고, 언어는 순전히 토글로만 제어" 라는 명시적 요구. 정적 파일이므로 런타임 API·키 노출이 없고, 페이지 이동 후에도 언어가 유지된다.
**트레이드오프**: 한 페이지에 두 언어가 모두 담겨 HTML이 다소 커지고, 검색엔진에 두 언어가 한 URL로 노출된다(개인 블로그·포트폴리오 규모에선 수용 가능). JS 비활성 시 한국어로 폴백.

### ADR-003: 폴더 기반 카테고리 + ko/en 짝 파일
**결정**: 글 = `<카테고리>/<slug>/` 폴더 + `ko.md`/`en.md`. 폴더가 곧 카테고리(중첩 가능)이자 언어중립 slug.
**이유**: 폴더만 추가하면 네비에 자동 노출 — 설정 파일 수정이 필요 없다. 한 글의 한/영 소스와 이미지가 한 폴더에 모여 관리가 쉽다.
**트레이드오프**: 파일명이 `ko.md`/`en.md`로 고정되는 관례를 지켜야 한다(`lib/content.ts`가 이 관례로 그룹핑).

### ADR-004: 폰트·글자 크기는 작성자 설정 전용(CSS 변수)
**결정**: `src/styles/tokens.css`의 CSS 변수 한 곳에서 폰트·크기·색상을 전역 제어. 방문자용 조절 UI는 만들지 않음.
**이유**: 사용자가 "작성자 설정만" 을 선택. 단일 소스라 변경이 쉽고 UI 복잡도가 없다.
**트레이드오프**: 방문자가 직접 글자 크기를 바꿀 수 없다(브라우저 확대/축소로 대체 가능).

### ADR-005: 영문 번역은 Claude가 생성하는 정적 파일
**결정**: 사용자는 `ko.md`만 작성하고, Claude가 `en.md`를 만들어 함께 커밋. 브라우저 실시간 번역 API는 쓰지 않음.
**이유**: 정적이라 빠르고 안전(키 노출 없음), 번역을 사람이 검토·수정할 수 있다.
**트레이드오프**: 번역이 자동 즉시 반영되지 않고 커밋 단계가 필요. `en.md`가 없으면 그 언어로 전환 시 "번역 준비 중" 안내 표시.

### ADR-006: 배포는 GitHub Actions
**결정**: 네이티브 Jekyll Pages 대신 `withastro/action` + `actions/deploy-pages` 워크플로로 배포.
**이유**: Astro는 빌드 단계가 필요하므로 Actions가 자연스럽다. `main` push만으로 빌드·배포가 자동화된다.
**트레이드오프**: Pages 소스를 "GitHub Actions"로 설정해야 하고 빌드 시간이 소요된다(수십 초 수준).
